## プロジェクト概要

**シフト管理・給与計算アプリ**（rtech / 代表: 守屋氏）

プログラミング教室・イラスト教室の2事業を個人経営しており、アルバイト従業員のシフトと給与を管理する Web アプリ。現状は従業員が各自 Excel に勤務時間・交通費を入力しているが、**改ざんが可能・手入力ミス・明細作成の手間**が課題。

**目的**: 管理者がシフトを一元管理し、シフトデータから給与を自動計算・明細を自動生成することで、**改ざん/ミスの排除**と**管理業務の省力化**を実現する。

- 要件定義書: `シフト管理アプリ要件定義書_v0_1.pdf`（ドラフト v0.1 / 2026-07-07）
- ステータス: **要件定義段階。コード・技術スタック未確定。**

## コアとなる設計原則（改ざん防止が中核）

これらはアプリの存在意義そのもの。実装時に絶対に崩さないこと。

1. **従業員は一切のデータ編集ができない** — 閲覧のみ。シフト修正は「修正依頼コメント送信」だけ（直接編集不可）。
2. **従業員は他人のシフト・給与を閲覧できない** — 給与額は個人情報。ロールベースアクセス制御（RBAC）を徹底。
3. **確定後のデータはロック** — 給与確定後のシフト・給与は編集不可。
4. **変更履歴を記録（監査性）** — シフト・実績の変更は「誰が・いつ・何を」変更したか記録する。

## ロール

| ロール | 対象 | 操作 |
|--------|------|------|
| 管理者 (admin) | 守屋氏 | シフト作成・編集、従業員管理、時給設定、実績修正、給与確定、明細発行、集計 |
| 従業員 (staff) | 各教室のアルバイト | 自分のシフト閲覧、自分の給与明細閲覧・PDF DL、修正依頼コメント送信 |

## ドメインの重要ルール（給与計算）

- **勤務実績 = 確定シフト**（打刻機能なし）。ただし欠勤・遅刻・早退・時間変更が実際に起こるため、**締め前に管理者が実績を確認・修正するフロー必須**。実績修正は管理者のみ、履歴を記録。
- **時給は「従業員 × 業務（教室）」の組み合わせ**で設定。例: Aさん = プログラミング講師 1,300円 / イラスト補助 1,100円。適用開始日を持ち、昇給履歴に対応。
- **交通費**: 従業員ごとに日額固定。`出勤日数 × 日額` で自動計算。適用開始日つき。
- **深夜割増**: 22:00〜翌5:00 は基礎時給の **25%増**（労基法準拠）。
- **残業割増**: 法定労働時間（1日8時間 / 週40時間）超過分は **25%増**。
- **基本給** = Σ(勤務時間 × 該当時給)、**割増** = 深夜 + 残業、**交通費** = 出勤日数 × 日額。
- 締め日ごとに自動集計 → 管理者が確認して「確定」→ 確定後ロック。

## 事業区分

- プログラミング教室 / イラスト教室 の2事業（Business）を区別して管理。
- 従業員の所属は「プログラミング / イラスト / 両方」。

## データモデル（概要）

| モデル | 内容 |
|--------|------|
| `User` | 氏名、メール、ロール(admin/staff)、在籍状態 |
| `Business` | 事業（プログラミング教室 / イラスト教室） |
| `WageRate` | 従業員 × 業務ごとの時給（適用開始日つき、昇給履歴対応） |
| `Shift` | 日付、開始/終了、休憩、従業員、業務、ステータス(下書き/公開/確定) |
| `ShiftRevision` | 実績修正の履歴（監査） |
| `CommuteAllowance` | 従業員ごとの交通費日額（適用開始日つき） |
| `Payroll` | 締め期間、従業員、各種金額、確定日時 |
| `PayrollDetail / PDF` | 明細内訳 |

シフトのステータス遷移: **下書き → 公開（従業員へ表示）→ 確定（ロック）**。

## 非機能要件

- **セキュリティ**: RBAC。従業員は自分のデータのみ閲覧可。認証はメール + パスワード。
- **監査性**: シフト・実績の変更履歴を記録。
- **データ保全**: 給与確定後は編集不可（ロック）。定期バックアップ。**データ消失は許容しない。**
- **可用性**: 個人事業規模のため高可用性は不要。
- **対応環境**: PC・スマホの主要ブラウザ（Chrome / Safari / Edge）。レスポンシブ UI。
- **規模**: 従業員 〜20名程度、同時アクセス少数。

## 追加機能（要件書 v0.1 以降に決定した変更）

要件定義書 v0.1 には未記載だが、モック検討の過程で追加が決まった機能。要件書の次版に反映すること。

### 源泉徴収票の発行（要件 v0.1「集計ダッシュボード」を置き換え）

- 画面7は「集計ダッシュボード」から **源泉徴収票の発行・ダウンロードページ** に変更。
- **確定した過年度分（例: 前年 = 2025年分）** の源泉徴収票を従業員ごとに PDF ダウンロードできる。全員分の一括DLも可。
- **本年度（進行中）の見込み版は発行しない** — 年度確定（年末調整後）にのみ発行する。
- 明細項目: 支払金額、課税対象額、源泉徴収税額（いずれも年度累計）。区分（甲欄/乙欄）を保持。
- **注意**: 源泉徴収票は税額計算を伴う。要件 v0.1 では「本格的な税計算はスコープ外」だったため、どこまで自動計算するか（甲乙欄・扶養控除・社会保険料控除・年末調整）は要決定。当面は簡易計算または確定額の手入力の可能性あり。

### 時間外勤務（在宅）申請の承認

- **シフト外に自宅等で行った勤務**。従業員が申請 → 管理者が承認/却下 → 承認分のみ給与に加算。
- 「従業員は直接編集不可・申請（コメント送信）のみ」という改ざん防止方針と整合（修正依頼と同じ申請系）。
- 法定残業（1日8h/週40h超過分の25%割増）とは **別物**。加算単価は原則その業務の通常時給（深夜/残業割増は付けない前提・要確認）。
- 承認/却下の履歴を記録（監査）。
